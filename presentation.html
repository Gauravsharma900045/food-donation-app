Introduction

What is FoodShare?

🎯

The Project

FoodShare is a full-stack web application that connects food donors — restaurants, families, events — with food receivers like NGOs, shelters, and individuals in need, all in real-time.

🌐

Live Deployment

  • Frontend hosted on Netlify
  • Backend hosted on Render.com
  • Database on MongoDB Atlas (cloud)
  • Accessible from any device, anywhere
⚠️

The Problem Statement

  • 68.7 million tonnes of food wasted in India annually (UNEP 2021)
  • Millions go to bed hungry every night across India
  • No easy, digital channel for surplus food to reach the underprivileged
  • Existing platforms lack real-time coordination
  • No photo-based listings or distance-based discovery

Code Explanation

Important Code Blocks

1️⃣ JWT Auth Middleware (auth.js)

function verifyToken(req, res, next) {
  const token = req.headers.authorization;
  if (!token) return res.status(401).send("Access Denied");
  try {
    const verified = jwt.verify(token, JWT_SECRET);
    req.user = verified; // { id, role }
    next();
  } catch (err) { res.status(400).send("Invalid Token"); }
}

Reads JWT from Authorization header, verifies cryptographic signature, attaches decoded { id, role } to req.user for downstream routes.

2️⃣ Distance Sort — Haversine Formula (food.js)

const calcDist = (lat1,lon1,lat2,lon2) => {
  const p = 0.017453292519943295; // π/180
  const a = 0.5 - Math.cos((lat2-lat1)*p)/2 +
    Math.cos(lat1*p) * Math.cos(lat2*p) *
    (1 - Math.cos((lon2-lon1)*p))/2;
  return 12742 * Math.asin(Math.sqrt(a)); // km
};
foods.map(f => { f.distance = calcDist(...); return f; })
     .sort((a,b) => a.distance - b.distance);

Spherical geometry calculation — same algorithm used by Google Maps. Foods sorted ascending — nearest first.

3️⃣ Food Schema (Food.js)

const FoodSchema = new mongoose.Schema({
  foodType: String, quantity: String,
  category: { type: String, enum:["veg","non-veg","both"] },
  status:   { type: String, default: "available" },
  image:    String,
  donor:    { type: ObjectId, ref: "User" },
  requestedBy: { type: ObjectId, ref: "User" },
  lat: Number, lng: Number
}, { timestamps: true });

ref:"User" enables Mongoose .populate() — like a SQL JOIN. Status field acts as state machine: available→requested→completed.

4️⃣ Environment Config (config.js)

const API_BASE =
  window.location.hostname === "localhost"
    ? "http://localhost:5000"       // Dev
    : "https://foodshare-backend-ao8p.onrender.com"; // Prod

5️⃣ Geolocation Fallback

navigator.geolocation?.getCurrentPosition(
  pos => fetch(`${url}?lat=${pos.coords.latitude}`),
  ()  => fetch(url) // user denied — load without GPS
);

Optional chaining ?. prevents crash if geolocation not supported. Two callbacks: success (with GPS) and failure (without GPS).

Challenges Faced

Real Engineering Challenges

1
CORS Errors — Cross-Origin Blocking
Frontend on Netlify and Backend on Render are on different domains. Browser blocks cross-origin API calls by default, breaking all data fetching.
2
Image Upload & Display in Production
Multer saves images to /uploads/ on Render server. Frontend URLs are different — relative paths broke all food images in production deployment.
3
JWT Token Persistence & Guards
Token must survive page reloads. Expired/invalid tokens must redirect to login. Needed guards on both frontend (redirect) and backend (middleware).
4
Geolocation: User Denial Handling
Not all users grant GPS access. App must sort by distance if available, load food normally if user denies — two completely different code paths.
5
Responsive Dashboard Sidebar
The 280px sidebar + main 2-column layout completely breaks on mobile. Needed a hamburger toggle, slide-in overlay, and bottom nav for phones.
6
Real-Time Notifications (Without WebSockets)
True real-time push requires WebSockets. Implemented on-demand fetching when bell is clicked — balances UX with server load.
7
Dev vs Production URL Management
API URL is http://localhost:5000 in development and https://render.com/... in production. Hardcoding either breaks the other environment.
8
Preventing Business Logic Violations
Donors requesting their own food, receivers completing donations, duplicate requests — all needed explicit server-side guards, not just frontend checks.
Solutions Implemented

How We Solved Each Challenge

✅ CORS Fix (server.js)

const cors = require("cors");
app.use(cors()); // Access-Control-Allow-Origin: *

Single line — adds CORS headers to every response before routes run.

✅ Image URL Fix (dashboard.html)

let imgSrc = img.startsWith('http')
  ? img
  : `https://foodshare-backend-ao8p.onrender.com
     /uploads/${img}`;
// onerror fallback to Unsplash CDN image

Detects if path is already absolute. Auto-prefixes production domain otherwise.

✅ JWT Route Guard

const token = localStorage.getItem("token");
if(!token) window.location.href = "login.html";

Every protected page checks for token on load. Server middleware independently validates it on every API call.

✅ Geolocation Fallback

navigator.geolocation?.getCurrentPosition(
  pos => fetchAndRender(
    `${url}?lat=${pos.coords.latitude}
     &lng=${pos.coords.longitude}`)
  ,() => fetchAndRender(url) // denied
);

Optional chaining ?. handles unsupported browsers. Two paths ensure the app always works.

✅ Business Logic Guards (food.js)

if(food.donor.toString() === req.user.id)
  return res.status(400).json({
    error: "Cannot request your own donation" });
if(food.status !== "available")
  return res.status(400).json({
    error: "Already requested" });

Server-side validation — cannot be bypassed by client-side manipulation. ObjectId compared as strings.

Performance & Optimization

Built for Speed & Security

⚡

API Performance

  • .lean() on read queries — skips Mongoose hydration, 30-40% faster
  • .populate("donor","name") — fetches only name field, not full user doc
  • DB query filtered: only status:"available" foods fetched
  • Distance sort done in Node, not DB (faster for small datasets)
🖼️

Image Optimization

  • Max 5MB per upload enforced by Multer
  • Only JPG, PNG, WEBP accepted (MIME type check)
  • Unique timestamped filenames — no collisions
  • CDN fallback (Unsplash) on image error
  • object-fit:cover — consistent display regardless of source dimensions
🔐

Security Hardening

  • bcrypt 10 rounds — 1024 iterations, brute-force resistant
  • JWT 7-day expiry — limits stolen token exposure
  • express.json({ limit:"10kb" }) — blocks oversized payload attacks
  • Server-side role checks — client cannot forge permissions
🎨

CSS Performance

  • CSS Variables (:root) — one change updates entire theme
  • GPU-accelerated transform for all animations (no layout reflow)
  • Staggered animation delays — prevents simultaneous reflows
  • backdrop-filter used sparingly — only navbar and modal
📱

Responsive Performance

  • Single HTML file per page — minimal HTTP requests

Viva Q&A

Expected Viva Questions

Q1. What is JWT and why use it instead of sessions?
JWT (JSON Web Token) is a cryptographically signed token encoding user identity. It's stateless — server doesn't store session data. Stored in localStorage, sent in Authorization header. Expires in 7 days for security.
Q2. How does bcrypt work? What are salt rounds?
bcrypt with 10 salt rounds performs 2^10 = 1024 hashing iterations. Cannot be reversed. On login, bcrypt.compare() rehashes the input and compares — timing-safe comparison prevents timing attacks.
Q3. What is the Haversine Formula?
A spherical geometry formula calculating the great-circle distance between two GPS coordinates on Earth. Used in /api/food/all to compute distance from user's GPS to each food listing, then sort ascending — nearest first.
Q4. What is Multer and how does file upload work?
Express middleware for multipart/form-data. Validates file MIME type (JPG/PNG/WEBP only), enforces 5MB limit, saves to /uploads/ with unique timestamped filename. Path stored in MongoDB as string.
Q5. What does .populate() do in Mongoose?
Replaces an ObjectId reference with the actual document from the referenced collection — like SQL JOIN. populate("donor","name") fetches only the name field from User, not the full document.
Q6. How does role-based access work?
Role (donor/receiver/admin) stored in JWT payload and MongoDB. Frontend reads from localStorage to show/hide UI. Backend middleware checks req.user.role on every protected route — server-side enforcement cannot be bypassed.
Q7. What is CORS and why was it needed?
Cross-Origin Resource Sharing — browser security policy blocking requests from different domains. Frontend (Netlify) and backend (Render) are on different domains. cors npm package adds Access-Control-Allow-Origin: * headers.
Q8. What happens when a user deletes their account?
Cascade deletion: (1) User document deleted, (2) All Food documents where donor matches user ID deleted, (3) All Notifications for that user deleted. Ensures no orphaned documents remain in the database.
Q9. Embedded vs Referenced documents in MongoDB?
We used referencing (ObjectId + populate) because User data is large and shared across many food documents. Embedding would duplicate user data in every food document — wasteful and hard to update consistently.
Q10. How does the geolocation fallback work?
navigator.geolocation?.getCurrentPosition(successCb, failCb). Optional chaining ?. handles unsupported browsers. failCb loads food without GPS params — app works normally. successCb sends lat/lng as query params for distance sorting.
